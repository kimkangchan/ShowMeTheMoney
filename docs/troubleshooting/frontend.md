# 프론트엔드 트러블슈팅

## 라이트/다크 모드 색상 하드코딩 문제

### 문제 상황

대시보드, 거래내역, 고정지출, 사이드바 등 여러 페이지에서 라이트 모드로 전환해도 배경/텍스트가 계속 어둡게 표시됨.

`dark:` 접두사 없이 `bg-gray-800`, `text-white` 같은 다크 전용 색상 클래스가 그대로 하드코딩되어 있어, 라이트 모드에서도 다크 색상이 무조건 적용되는 것이 원인.

### 해결

- 다크 모드 전용 스타일에는 반드시 `dark:` 접두사를 붙이도록 수정
- 영향 받은 파일: `dashboard/page.tsx`, `transactions/page.tsx`, `recurring/page.tsx`, `components/layout/Sidebar.tsx`
- 예산 진행바, 차트, 카드, 필터 버튼, 테이블, 구분선 등 컴포넌트별 색상 전수 점검

### 결론 / 배운 점

- 다크 모드를 기본값처럼 생각하고 스타일을 짜면 라이트 모드가 항상 깨짐
- 색상 클래스를 추가할 때마다 라이트/다크 두 모드 모두에서 확인하는 습관 필요

## 로그인 401 응답 시 무한 리다이렉트 루프

### 문제 상황

토큰 만료(401) 시 로그인 페이지로 강제 이동시키는 로직이, 이미 로그인 페이지에 있는 상태에서도 계속 발동되어 새로고침이 반복되는 무한 루프 발생.

### 원인

`lib/api.ts`의 axios 응답 인터셉터에서 401을 감지하면 무조건 `/auth/login`으로 리다이렉트하도록 구현되어 있었음. 로그인/회원가입 페이지 자체에서 발생하는 401(예: 로그인 실패)에도 동일하게 반응하면서 리다이렉트 → 페이지 재요청 → 401 → 리다이렉트가 반복됨.

### 해결

- 인터셉터에서 리다이렉트 전에 현재 경로가 `/auth`로 시작하는지 확인하는 조건 추가

```ts
if (error.response?.status === 401) {
  localStorage.removeItem("accessToken");
  if (!window.location.pathname.startsWith("/auth")) {
    window.location.href = "/auth/login";
  }
}
```

### 결론 / 배운 점

- 전역 인터셉터에 리다이렉트 로직을 둘 때는 "이미 그 페이지에 있는 경우"를 항상 예외 처리해야 함

## 로그인/회원가입/모달 흰 배경에 흰 글자

### 문제 상황

로그인, 회원가입, 거래 등록 모달 페이지에서 입력한 텍스트가 배경과 같은 흰색으로 표시되어 사실상 보이지 않음.

### 원인

카드 컨테이너(`bg-white`)에 텍스트 색상을 지정하지 않아 상위 레이아웃(다크 배경 기준)의 흰색 텍스트 스타일이 그대로 상속됨.

### 해결

- 카드 컨테이너에 `text-gray-900`을 명시적으로 추가

```tsx
<div className="bg-white rounded-2xl shadow-md w-full max-w-sm px-8 py-10 text-gray-900">
```

### 결론 / 배운 점

- 흰 배경 카드를 다크 테마 레이아웃 안에 배치할 때는 텍스트 색상을 컨테이너 단위로 명시해야, 상위에서 상속되는 흰색 글자에 뒤덮이지 않음

## 프론트-백엔드 데이터 타입 불일치 (카테고리 UUID, isActive 등)

### 문제 상황

고정지출(recurring) 기능에서 카테고리 선택, 활성/비활성 토글, 삭제 등이 백엔드와 정상 연동되지 않음.

### 원인

- 프론트는 카테고리를 `categoryUuid`(number)로 다뤘지만, 백엔드는 `categoryCode`(string)를 기대함
- `isActive`를 프론트는 `1`/`0` 정수로, 백엔드는 `boolean`으로 주고받아 토글 요청이 어긋남
- API 경로/식별자도 불일치 (`/api/recurring` → `/api/recurring-items`, `uuid` → `id`)

### 해결

- 프론트 타입을 백엔드 응답 구조에 맞춰 `categoryCode`(string) 기준으로 통일
- 토글 요청 시 `isActive`를 `boolean`으로 변환해서 전송
- API 경로와 리소스 식별자(`id`)를 백엔드 스펙에 맞게 수정

### 결론 / 배운 점

- 프론트/백엔드를 병행 개발할 때 응답 필드명·타입을 초기에 명확히 합의하지 않으면 통합 단계에서 자잘한 불일치가 여러 곳에서 동시에 터짐
- `types/index.ts`로 타입을 중앙 관리하되, 백엔드 DTO가 바뀌면 즉시 동기화하는 프로세스 필요

## API URL 하드코딩으로 배포 환경에서 백엔드 접근 불가

### 문제 상황

로컬 개발 환경에서는 문제없이 동작했지만, Docker/K8s 등 외부 환경에 배포하면 프론트엔드가 백엔드 API를 호출하지 못함.

### 원인

API 요청 baseURL이 `http://localhost:8080`으로 고정되어 있어, 프론트엔드가 브라우저에서 실행될 때 로컬이 아닌 배포 환경(다른 호스트/포트)에서는 접근이 불가능했음.

### 해결

- `lib/api.ts`에서 baseURL을 환경변수로 분리

```ts
const api = axios.create({
  baseURL: process.env.NEXT_PUBLIC_API_URL ?? "http://localhost:8080",
  ...
});
```

- 로컬은 기본값(`localhost:8080`) 사용, 배포 환경은 `NEXT_PUBLIC_API_URL`로 실제 접근 가능한 백엔드 주소 주입

### 결론 / 배운 점

- CORS 허용 origin과 마찬가지로, 프론트엔드가 호출하는 백엔드 주소도 환경마다 달라질 수 있는 값이므로 하드코딩하지 않고 환경변수로 분리해야 함
- `NEXT_PUBLIC_` 접두사가 붙은 환경변수만 브라우저(클라이언트) 코드에서 접근 가능하다는 Next.js 특성에 주의

### 기술 스택

- **Frontend**: Next.js, React, TypeScript, axios
